# 11. Спецификация требований

**Спецификация требований ПО** - структурированный набор требований (функциональность, производительность, конструктивные ограничения и атрибуты) к ПО и его внешним интерфейсам. Предназначен для того, чтобы установить базу для соглашения между заказчиком и разработчиком (или подрядчиками) о том, как должен функционировать программный продукт.

## Что включает в себя спецификация требований?
1. Первичный список требований (функциональные и нефункциональные)
2. Модели требований (анализ и детализация требований)
    - Модель предметной области. Объекты и взаимосвязи между ними. Возможно модели данных для этих объектов. Рамки разрабатываемой системы
    - Модели пользователей системы
    - Функциональная модель. Сценарии использования системы или детальное описание работы каждой функции с точки зрения пользователя
3. Высокоуровневая архитектура системы
    - Модель архитектуры
    - Детальные требования к основным подсистемам
4. Критерии аттестации системы
    - Набор базовых высокоуровневых тестов и характеристик, которые будут проверяться при аттестации ПО и удрстоверяют его соответствие требованиям заказчика
5. Глоссарий терминов

## Содержание спецификации требований

1. Пользовательские требования - первичный список требований
2. Модель (система) требований.
3. Высокоуровневая архитектура ПС
4. Критерии аттестации - не требования, а некие планы, как заказчику будет сдаваться проект. Если эти критерии выполнены, то заказчик увидит, что он получил то, что хотел. Как правило, сценарии высокоуровневые и должны включать все функции из пользовательских требований. Могут быть и сложные варианты сценариев.

> Модель требований во множественном числе, так как если много требований, то каждая система будет описывать группу системных требований (модели по функциям из пользовательских требований)
