# 16. Верификация и аттестация: инспектирование и отладка

::: details Основная информация

Процессы проверки и анализа, в ходе которых оценивается качество разрабатываемой ПС на основе соответствия требованиям заказчика.  
В первую очередь качество - сопоставлениес требованиями (насколько они выполняются).

**Верификация** - правильно ли разработана ПС (соответствие трелбованиям).

**Аттестация** - правильно ли работает ПС (соответствие ожиданиям заказчика).
:::

## Инспектирование

Статическая проверка различных представлений ПС.
- Проверка спекификации требований
- Проверка архитектуры
- Анализ исходного программного кода
- Проверка документа проектирования
- Проверка плана проекта

### Преимущества и недостатки

|Преимущества|Недостатки|
|---|---|
|Запуск ПС часто является дорогостоящим процессом, а тут он не требуется -> минимальные затраты :)|Требует времени на подготовку, проведение и анализ результатов от ключевых специалистов, что дорого :( |
|За один сеанс инспектирования можно найти несколько ошибок и подозрительные  места|Не может найти ошибки, связанные с производительностью, взаимодействием с окружением, временными условиями|
|Можно сосредоточиться на конкретных типах ошибок (т.е. ищем синтаксические ошибки в коде или подозрительные к утечкам памяти иил производительности части)|При длительном инспектировании (более 1-2 часов) концентрация внимания падает, и эффективность поиска дефектов резко снижается.|

### Типы ошибок
1. Полнота (спецификация, архитектура)
2. Непротиворечивость (документы)
3. Визуальное оформление (документы, код)
4. Стиль кодирования (корпоративный стиль кода - брендбук)
5. Трассируемость требований (какие требования учтены и в какой степени)

> Основные объекты - документация и программный код.

### Дополняющие методы
- Программный инструмент - автоматический статический анализ кода (компилятор, например)
- Формальные методы (математические)
- Система, наблюдающая за работой пользователя (анализ движения зрачков для проверки интерфейса, переключение между мышью и кклавиатурой)
- ИИ-инспекторы и ИИ-корректор кода

### Процедура
Формализованный процесс:
- Автор (кода или документа)
- Инспектор(ы)
- Председатель инспекционной группы
- Секретарь (для ведения документации)

Шаги:
1. Цель и объект (часть кода на предмет стиля)
2. Инспектор(ы) анализируют объект и формируют список дефектов (не более 1-3 часа)
3. Собрание группы (не более 1 часа) - определить, какие дефекты являются ошибками и включить их в план отладки

<span style="color: red"> НА СОБРАНИИ НЕ ОБСУЖДАЕТСЯ, КАК ИСПРАВЛЯТЬ ОШИБКУ!!! </span>

4. Назначаются ответственные по исправлению ошибок из плана отладки

## Методы исправления ошибок - отладка:
- "Чистая комната" - утранение ошибок по ходу разработки
- "Генеральная уборка" - устранение ошибок по окончании разработки

> Цель - разработка ПО и составление документации без дефектов.
