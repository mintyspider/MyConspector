# 10. Функциональные и нефункциональные требования

> В ходе заполнения этой статьи использовались данные с лекции по [ПИС](/disciplines/5semester/pis/index) от 24/10/2025.

## Функциональные требования

Перечень функций (услуг, сервисов), которые должна выполнять ПС.  

::: details Оффтопик: что входит в функциональные требования?
**Бизнес-требования** -  
определяют назначение ПО, описываются в документе о видении и границах проекта. Если проще, то почему организации нужна такая система, то есть цели, которые огранизация намерена достичь с ее помощью. Например,

+ Увеличение продаж
+ Снижение затрат на ремонт
+ Оптимизация затрат на персонал

**Пользовательские требования** -  
определяют набор пользовательских задач, которые должна решать программа, а также способы (сценарии) их решения в системе. Если проще, то описываются цели или задачи, которые пользователи должны иметь возможность выполнять с помощью продукта, который в свою очередь должен приносить пользу кому-то. Например,

+ Формирование заказов на продукцию
+ Определение степени износа узлов
+ Контроль действий пользователя 

Сами **функциональные требования** -  
определяют, каким должно быть поведение продукта в тех или иных условиях, что разработчики должны создать, чтобы пользователи смогли выполнить свои задачи (пользовательские требования) в рамках бизнес-требований. Например,

+ Система должна добавлять товар в корзину
+ Система должна уведомлять оператора о выявленных дефектах
+ Система должна проверять местоположение пользователя каждые 5 минут
:::

Также должно быть указано, как система реагирует на:
+ Входные данные (в систему поступают данные -> система реагирует)
+ Поведение в различных ситуациях (место кончилось, сбой в системе и др.)
+ что ПС ***не*** должна делать (в явном виде, чтоб заказчик понимал, чего нет в проекте)

Шаблоны оформления:

```markdown
Пользователь должен иметь возможность <название функции>.
```

```markdown
Программная система должна предоставлять <название функции>.
```

```markdown
[Роль|группа] должен(на) иметь возможность <выполнить что-то>  
<с каким-то объектом> <при условии [время отклика|критерий качества]>.
```
```markdown
<Предварительное условие> при <триггере события> программная система должна  
<как-то (ожидаемо) отреагировать на событие>.
```

## Нефункциональные требования

Ограничения на выполняемые функции из функциональных требований.

::: details Оффтопик: Что включают в себя нефункциональные требования?
Бизнес-правила:
- включют корпоративные политики, правительственные постановления, отраслевые стандарты и вычислительные алгоритмы
- они часто налагают ограничения, определяя, какими функциями должна обладать система, подчиняющаяся соответствующим правилам
- становятся источником атрибутов качества

Атрибуты качества:
- представляют собой описание различных измерений характеристик продукта, которые важны для пользователей или для разработчиков и тех, кто будет обслуживать систему, таких как производительность, доступность, применимость, надежность, переносимость и эксплуатацонная пригодность

Внешние интерфейсы:
- правила подключения к другим программным системам, аппаратным устройствам и пользователям, а также коммуникационные интерфейсы

Ограничения
- границы на возможности выбора разработчика при проектировании продукта

 :::

### Виды ограничений
+ на производительность (скорость получения ответа и т.п.)
+ предметной области (формат данных и особенности каждого формата)
+ технические средства и технологии программирования
+ интерфейс пользователя (UI), внутренние интерфейсы, взаимодействия с внешними системами (память, передача данных)
+ интеграционные свойства ПС

### Основные группы
```
Атрибуты качества:
- удобство использования
- производительность
- безопасность
- защита
- надежность
- совместимость
- масштабируемость

Внешние интерфейсы:
- интерфейс пользователя
- программные интерфейсы
- интерфейсы оборудования
- коммуникационные интерфейсы

Ограничения
```

### Области появления нефункциональных требований
+ информационная безопасность (приватные данные, шифрование данных, техпроцессы)
+ производительность (время обработки данных, скорость обработки заявок)
+ структуры данных (формат данных и методы работы с ними)
+ законодательство (распространение и использование, лицензия, права на код)
+ организационные требования (расписание работ, встречи с заказчиком)

## Качественные требования

Характеристики качественных требований по-разному определены различными источниками.

- Единичность - требование описывает одну и только одну вещь.
- Завершенность - требование полностью определено в одном месте и вся необходимая информация присутствует
- Последовательность - требование не противоречит другим требованиям и полностью соответствует внешней документации
- Атомарность - требование не может быть разбито на ряд более детальных требований без потери завершенности
- Отслеживаемость - требование полностью или частично соответствует деловым нуждам как заявлено заинтересованными лицами и документировано
- Актуальность - требование не стало устаревшим с течением времени
- Выполнимость - требование может быть реализовано в пределах проекта
- недвусмысленность - требование кратко определено без обращения к техническому жаргону, акронимам и другим скрытым формулировкам. Оно выражает объективные факты, а не субъективные мнения. Возможна одна и только одна интерпретация. Определение не содержит нечётких фраз. Использование отрицательных и составных утверждений запрещено.
- Обязательность - требование представляет определенную заинтересованным лицом характеристику, отсутствие которой приведет к неполноценности решения, которая не может быть прогнозированна.
    - необязательное требование - противоречие самому понятю требования
- Проверяемость - реализованность требования может быть определена через один из 4 возможных методов:
    - осмотр
    - демонстрация
    - тест
    - анализ

## Количественные показатели

Затруднительно сформулировать ограничения таким образом, чтобы проверить их выполнение. Поэтому в формулировках требований необходимо использовать количественные показатели, например:

|Показатель|Измерение|
|---|---|
|Скорость работы|транзакции/время/время реакции на действие пользователя/время обновления экрана|
|Размер ПС|максимальное число записей в БД/максимальный размер установочного файла/количество модулей (у параллельных программ)|
|Простота эксплуатации|время обучения пользователя/количество статей в справочной системе|
|Надёжность (работа до сбоя)|среднее время безаварийной работы/вероятность выхода системы из строя в зависимости от условий эксплуатации|
|Устойчивость к сбоям (восстановление)|время восстановления системы/процент событий, приводящих к сбоям/вероятностьпотери данных из-за сбоя/объем потерянных данных|
